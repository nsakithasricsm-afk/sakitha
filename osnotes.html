Overview of Operating Systems

The foundational concepts every OS exam begins with

Core Concept
An Operating System (OS) is system software that acts as an intermediary between computer hardware and application programs — managing resources and providing a convenient, safe environment for program execution. It is simultaneously a resource allocator (decides who gets CPU, memory, I/O) and a control program (prevents errors and misuse by user programs).
Key Terminology
Kernel
Core of the OS, always resident in memory; manages CPU, memory, and devices
Shell
User interface to the kernel — CLI (bash, zsh) or GUI (Windows Explorer, GNOME)
Process
A program in active execution; has its own memory space, registers, and state
Interrupt
Hardware/software signal that causes the CPU to pause and handle an urgent event
Dual-mode Operation
CPU has user mode (restricted) and kernel mode (privileged) — hardware-enforced protection
Bootstrapping
Power-on sequence: BIOS/UEFI → POST → Bootloader → Kernel loaded into RAM
Multiprogramming
Multiple programs kept in memory; CPU switches to another when current job does I/O
Time-sharing
Rapid CPU switching among users (time quantum ~100ms) giving illusion of dedicated machine
Spooling
Simultaneous Peripheral Operations On-Line — buffers I/O jobs (e.g. print queue)
System Call
Formal API for user programs to request kernel services (file I/O, process creation, etc.)
Mechanism / Process — How the OS boots & runs
1
Power ON → BIOS/UEFI runs POST (Power-On Self Test), detects RAM/CPU/storage
2
Bootloader (GRUB / Windows Boot Manager) found on disk and executed
3
OS kernel loaded into RAM; hardware device drivers initialised
4
Process scheduler, memory manager, and file systems mounted and ready
5
init / systemd (Linux) or Session Manager (Windows) starts daemons and services
6
Login prompt displayed → user authenticates → shell/GUI launched
7
User programs run in user mode; cross to kernel mode via system calls only
Pros & Cons
FeatureAdvantageDisadvantage
AbstractionHides hardware complexity from developersAbstraction layers add overhead
Resource SharingMultiple users/apps share hardware efficientlyScheduling contention and complexity
Security (Dual-mode)User programs cannot access raw hardwareComplex attack surface on the OS kernel itself
PortabilityApps written for the OS run on any supported hardwareOS must be ported for each new hardware architecture
Error HandlingOS catches and handles hardware/software errorsCan mask bugs; error handling adds code complexity
Example — Time-sharing with 4 users
Setup: 4 users (A, B, C, D) share one CPU. Time quantum = 100ms.

Execution order:
  A runs 100ms → B runs 100ms → C runs 100ms → D runs 100ms → A again...

Key insight: When A waits for keyboard input (I/O blocked), the OS
immediately gives the CPU to B. CPU is never idle while work is pending.

Without OS: A locks CPU for 2 hours; B, C, D wait.
With time-sharing OS: All 4 users respond in < 400ms — feels instant.

Chef analogy: One chef (CPU) serves 4 guests. While guest A's cake
bakes (I/O wait), the chef chops vegetables for guest B. No idle time.
Exam Pitfalls
✕
OS ≠ Kernel. The kernel is just the core. The full OS includes utilities, shell, libraries, and system programs.
✕
Multiprogramming ≠ Multiprocessing. Multiprogramming = many programs on ONE CPU. Multiprocessing = multiple CPUs. Never swap these.
✕
Time-sharing IS a form of multiprogramming — it adds user responsiveness on top. They are not opposites.
✕
Only the kernel is always in memory. User programs, shell, and utilities are swapped in/out as needed.
✕
BIOS/UEFI is firmware, NOT the OS. It runs before the OS loads. The OS is stored on disk; firmware is in ROM/flash.
✕
Interrupts are how the OS regains CPU control from running processes — not periodic checks or polling.

Types of Operating Systems

Classification by resource management strategy and use-case

Core Concept
OS types are classified by how they manage resources and serve users — from simple batch systems with zero interaction, to sophisticated distributed and real-time systems that must meet strict deadlines. Each type makes different trade-offs between throughput, responsiveness, reliability, and complexity.
Key Terminology
Batch OS
Groups similar jobs together; no user interaction during execution; maximises throughput
Time-sharing OS
Multiple interactive users share CPU via rapid context switching (time quantum)
Hard Real-time OS
Missing a deadline = system failure. Used in safety-critical systems (ABS, pacemakers)
Soft Real-time OS
Deadlines preferred but occasional misses tolerable (video streaming, phone calls)
Distributed OS
Multiple networked machines appear as one unified system to users
Network OS
Adds file sharing/printing over network; each machine keeps its own independent OS
Embedded OS
Stripped-down OS for dedicated hardware: routers, microwaves, car ECUs
Mobile OS
Touch-optimised, battery-aware OS for smartphones and tablets
Clustered OS
Multiple machines sharing storage for high availability and fault tolerance
Multiprocessor OS
Manages systems with multiple CPUs — symmetric (SMP) or asymmetric (AMP)
Mechanism / Process — Decision flow for selecting OS type
1
Batch OS: Collect jobs → group by resource type → submit as a job deck → run unattended → print output. No user interaction possible mid-execution.
2
Time-sharing: Allocate fixed time quantum (e.g. 100ms) → CPU runs process → timer interrupt fires → context switch → next process → repeat round-robin.
3
RTOS: Assign priorities to tasks → highest-priority ready task always runs → scheduler guarantees CPU within deadline window → interrupt latency bounded.
4
Distributed OS: Location-transparent resource allocation across network nodes → processes migrate to least-loaded node → single system image to user.
5
Embedded OS: ROM-based kernel → deterministic boot in milliseconds → minimal footprint → hardware-specific drivers only, no general-purpose extras.
Comparison Table
TypeBest ForLimitation
Batch OSHigh-throughput repetitive jobs (payroll, scientific computing)No interactivity; long turnaround time
Time-sharing OSInteractive multi-user environmentsContext-switch overhead; complex scheduling
Hard RTOSSafety-critical systems (aircraft, medical devices)Not general purpose; limited feature set
Distributed OSScalable cloud and scientific computingComplex design; network failure = problem
Mobile OSTouch devices, always-on, battery-constrainedLess capable than desktop OS
Embedded OSDedicated hardware with fixed functionVery limited; not user-programmable
Real-World Examples
Batch OS:
  University grade calculation runs nightly at 2 AM.
  500 student records processed → no interactive prompt → results emailed by 6 AM.

Hard Real-Time OS:
  ABS (Anti-lock Braking System) must respond to wheel-speed sensor in < 5ms.
  FreeRTOS on microcontroller guarantees this every time.
  Windows 11 is fast but NOT hard real-time — it cannot guarantee 5ms.

Distributed OS:
  Google's Borg system manages millions of containers across data centers.
  A job submitted from India may run on servers in Singapore — transparently.

Mobile OS:
  Android (Linux-based) manages battery, radio power, touch events.
  Screen-off → CPU throttles to 200MHz → saves 80% battery vs full speed.
Exam Pitfalls
✕
Network OS ≠ Distributed OS. In a Network OS, each machine keeps its own OS. In a Distributed OS, all machines appear as one unified system.
✕
Hard real-time ≠ just fast. Windows can respond in 1ms sometimes, but it still fails the hard real-time test because it cannot GUARANTEE the deadline every time.
✕
Embedded OS ≠ no OS. Most embedded systems run a real RTOS (FreeRTOS, VxWorks, Zephyr) — they just don't have a GUI or general-purpose features.
✕
Android is Linux-based but IS a full OS — not a "lite version." It has a full kernel, process management, memory management, etc.
✕
Batch OS has ZERO user interaction during job execution. If a question says "user can give input while running," it is NOT a batch OS.

OS Structures

How the OS organises its internal components

Core Concept
OS structure defines how kernel components are organised and communicate internally — from a single monolithic block where everything runs in kernel space, to a microkernel where only IPC and scheduling stay in the kernel and all other services run as isolated user-space processes.
Key Terminology
Monolithic Kernel
All OS services (scheduling, file systems, drivers) compiled into one large binary in kernel space
Layered OS
OS divided into numbered layers; Layer N may only use services from Layer N-1
Microkernel
Minimal kernel (IPC + basic scheduling); file systems, drivers run as user-space server processes
Modular OS
Core kernel + dynamically loadable kernel modules (LKMs) added at runtime — used by Linux
Hybrid Kernel
Combines monolithic performance with microkernel features (Windows NT, macOS XNU)
Exokernel
Exposes raw hardware resources directly to applications; maximum performance, minimum abstraction
HAL
Hardware Abstraction Layer — isolates OS from hardware-specific details (used in Windows NT)
IPC
Inter-Process Communication — how microkernel servers talk to each other (message passing)
Mechanism / Process
1
Monolithic: All services compiled into one binary. A file system call goes directly to the kernel function — no IPC, very fast, but a bug in any component can crash everything.
2
Layered: Layer 0 = hardware abstraction. Layer 1 = CPU scheduling. Layer 2 = memory management. ... Layer N = user interface. Each layer only calls the one directly beneath it.
3
Microkernel message flow: User app → kernel IPC → file server process → kernel IPC → storage driver process → kernel IPC → user app. More steps, but each server is isolated.
4
Modular (Linux): Boot with minimal kernel → user runs insmod wifi_driver.ko → module loaded into kernel space at runtime → rmmod to unload cleanly.
5
Hybrid (macOS XNU): Mach microkernel handles IPC + VM. BSD layer handles POSIX API. Both run in kernel space but are logically separate — best of both architectures.
Structure Comparison
StructureAdvantageDisadvantage
MonolithicHigh performance — no IPC overhead between componentsOne buggy driver crashes the entire OS
LayeredClean design; each layer debugged independentlyRigid hierarchy; hard to define layers correctly
MicrokernelFault isolation — driver crash doesn't crash kernelIPC overhead; slower than monolithic
ModularFlexible — load/unload components on demandModule bugs still run in kernel space and can crash OS
HybridBalance of performance and stabilityComplex to design and maintain
Example
Linux = Monolithic + Modular (not a pure microkernel!)
  One kernel binary. Device drivers as loadable modules.
  A buggy Wi-Fi driver (LKM) CAN crash Linux — kernel panic.

macOS XNU = Hybrid (Mach + BSD)
  Mach microkernel: IPC, virtual memory, task management
  BSD layer: POSIX API, file systems, networking
  Both compiled together — looks monolithic, built modularly.

Minix 3 = True Microkernel
  Kernel: 13,000 lines — only IPC + interrupt handling
  File system: runs as a user process
  If the file server crashes, the kernel restarts it automatically.
  Result: self-healing OS used in Intel Management Engine chips.
Exam Pitfalls
✕
Linux is NOT a microkernel. It is monolithic (with loadable modules). This is the #1 most common exam mistake on this topic.
✕
Layered OS layers are strictly ordered. Layer N can ONLY call layer N-1. It cannot skip to N-3 — this violates the design.
✕
Microkernel IS slower due to IPC overhead — this is a real, well-documented trade-off, not a myth.
✕
Loadable modules (LKMs) run in kernel space — they are NOT user-space processes. A buggy module can still crash Linux.
✕
Windows NT is hybrid, not monolithic. It uses a HAL and runs some subsystems outside the core kernel executive.

OS Services

What the OS provides to programs and users

Core Concept
OS services are the set of functions the OS provides to user programs and system administrators — covering program execution, file management, I/O operations, communications, error detection, resource allocation, accounting, and protection/security.
Key Terminology
Program Execution
OS loads program into memory, executes it, and handles normal/abnormal termination
I/O Operations
Manages access to files and devices; user programs cannot access hardware directly
File-System Manipulation
Create, delete, read, write, search files and directories; manage permissions
Communications (IPC)
Shared memory or message passing between processes on same or different machines
Error Detection
Detects and responds to CPU errors, memory faults, I/O failures, user program errors
Resource Allocation
Distributes CPU, memory, I/O ports, files among competing processes fairly
Accounting
Tracks which users consume which resources — for quotas, billing, or optimisation
Protection
Controls access to resources; ensures processes don't interfere with each other
Security
Authentication, encryption, defence against external attacks and unauthorised access
User Interface
CLI (bash, cmd) or GUI (GNOME, Windows) — the visible face of OS services
Mechanism / Process — Tracing one shell command through 5 services
1
User types: cat report.txt | grep "error" | wc -l
2
Process Execution: OS spawns 3 processes (cat, grep, wc) — one per command
3
File Service: Opens report.txt on behalf of cat; verifies read permission
4
IPC Service: Creates 2 anonymous pipes — cat's stdout → grep's stdin → wc's stdin
5
Resource Allocation: Scheduler gives each of the 3 processes CPU time fairly
6
I/O Service: wc writes final count to terminal (stdout file descriptor 1)
7
Process Exit: OS cleans up all 3 processes, releases memory, closes file descriptors
Services: User-facing vs System-facing
ServiceHelps User/ProgramOverhead/Trade-off
Program ExecutionHandles loading, linking, memory setup automaticallyVirtual memory management overhead
File SystemPersistent named storage with permissionsDisk I/O latency; journaling overhead
IPCProcesses cooperate safely without sharing raw memoryMessage copying overhead vs direct memory access
Protection & SecurityPrevents accidental and malicious damagePermission checks add latency to every operation
AccountingEnables fair quotas and billingLogging every resource use consumes CPU/disk
Exam Pitfalls
✕
Services ≠ System Calls. Services are WHAT the OS provides. System calls are HOW a program requests those services.
✕
Protection and Security are two different services. Protection = internal (processes can't interfere with each other). Security = external (authentication, defence against attacks).
✕
Accounting ≠ financial accounting. In OS context it means tracking resource usage (CPU time, disk space, I/O operations) per user or process.
✕
The User Interface (CLI/GUI) is an OS service — but it is NOT part of the kernel. The shell runs in user mode.
✕
Error detection ≠ error correction. The OS detects errors and reports or terminates the offending process — it does not necessarily fix the underlying hardware error.

System Calls

The API between user programs and the OS kernel

Core Concept
A system call is the programmatic interface by which a user-space process requests a privileged kernel service, triggering a CPU mode switch from user mode to kernel mode via a hardware trap instruction. It is the only legal way for user programs to access hardware or OS resources.
Key Terminology
System Call
Formal request from user process to OS kernel for a privileged service
Trap / Software Interrupt
CPU instruction (INT 0x80, SYSCALL) that switches from user mode to kernel mode
System Call Number
Integer placed in a register to identify which kernel function to call
System Call Table
Kernel's array of function pointers — sys_call_table[number] → handler function
POSIX
Portable OS Interface — standard defining Unix-like system call APIs (Linux, macOS comply)
libc / glibc
C standard library that wraps system calls in convenient functions (fopen → open syscall)
errno
Global variable set on failed system calls indicating error type (ENOENT, EACCES, etc.)
strace
Linux diagnostic tool that intercepts and logs all system calls made by a process
Context Switch
Saving/restoring CPU register state when switching between processes or privilege levels
fork()
Creates a child process — returns child PID in parent, returns 0 in child, -1 on error
Mechanism — Anatomy of one system call
1
User program calls fopen("file.txt", "r") — a C library (libc) function
2
libc puts system call number (e.g. 2 = open) in register RAX; args in RDI, RSI, RDX
3
libc executes SYSCALL instruction → CPU switches to kernel mode automatically
4
CPU saves user-mode registers to kernel stack → reads RAX → looks up sys_call_table[2]
5
Kernel handler runs: validates path, checks permissions, allocates file descriptor
6
Return value (file descriptor integer or -1) placed in RAX; CPU switches back to user mode
7
libc reads RAX → if -1, sets errno to indicate error → returns fd or NULL to caller
5 Categories of System Calls
CategoryExamplesWhat They Do
Process Controlfork, exec, exit, wait, killCreate, run, terminate, and synchronise processes
File Managementopen, read, write, close, stat, unlinkCreate, access, modify, delete files
Device Managementioctl, read, write (device files)Request/release devices; read/write device data
Information Maintenancegetpid, alarm, sleep, timeGet/set system time, process info, system data
Communicationpipe, socket, shmget, mmap, send, recvIPC via pipes, sockets, or shared memory
Example — strace on ls
$ strace ls 2>&1 | head -20

execve("/bin/ls", ["ls"], envp)          ← load and run ls binary
brk(NULL)                                ← check current heap end
mmap(...)                                ← map memory for dynamic linker
openat(AT_FDCWD, ".")                    ← open current directory
getdents64(3, /* 12 entries */, 32768)   ← read directory entries
write(1, "file1.txt  file2.py\n", 20)   ← print to terminal (stdout)
close(3)                                 ← close directory file descriptor
exit_group(0)                            ← terminate with success code 0

Observation: even a simple 'ls' makes ~80-100 system calls.
printf("hello\n") → triggers write() system call internally.
The user never sees the mode switches — libc handles them transparently.
Exam Pitfalls
✕
printf() is NOT a system call. It is a C library (libc) function that eventually calls the write() system call. This distinction appears in almost every OS exam.
✕
System calls involve a CPU mode switch (user → kernel → user). Regular function calls do NOT switch modes. The trap instruction is what makes it a system call.
✕
fork() returns TWICE. In the parent process it returns the child's PID (>0). In the child process it returns 0. If fork fails, it returns -1 only once (in the parent only).
✕
A failed system call returns -1 and sets errno. It does NOT throw a C++ exception. You must check the return value and errno manually.
✕
POSIX is a standard, not an OS. Linux, macOS, and others implement POSIX. Windows does not natively comply with POSIX.

Virtual Machines

Running multiple OS instances on one physical host

Core Concept
A Virtual Machine (VM) creates an isolated software emulation of complete computer hardware, allowing multiple guest OS instances to run simultaneously on a single physical host — managed by a hypervisor (VMM) that allocates physical resources to each VM.
Key Terminology
Hypervisor / VMM
Software layer that creates, runs, and manages virtual machines (Virtual Machine Monitor)
Type 1 Hypervisor
Runs directly on bare-metal hardware — no host OS. Examples: VMware ESXi, Xen, Hyper-V, KVM
Type 2 Hypervisor
Runs on top of a host OS as an application. Examples: VirtualBox, VMware Workstation, QEMU
Guest OS
The OS running inside a virtual machine (e.g. Ubuntu inside VirtualBox on Windows)
Host OS
The OS on the physical machine running a Type 2 hypervisor
Full Virtualisation
Guest OS runs completely unmodified; hypervisor traps and emulates privileged instructions
Para-virtualisation
Guest OS is modified to be aware it's in a VM — calls hypercalls instead of native hardware instructions for better performance
Container
Lightweight OS-level virtualisation; shares the HOST kernel (Docker, LXC) — not a full VM
Snapshot
Saved state of a VM at a point in time — enables rollback and rapid cloning
Live Migration
Moving a running VM between physical hosts with zero downtime (requires shared storage)
Mechanism / Process
1
Hypervisor installed on bare metal (Type 1) or on host OS (Type 2)
2
Admin defines VM: allocate 4 vCPUs, 8 GB RAM, 100 GB virtual disk, 1 virtual NIC
3
Guest OS installs onto virtual disk exactly as on real hardware — sees a fake but complete machine
4
Guest executes privileged instruction → hypervisor traps it → emulates result → returns to guest
5
Hypervisor scheduler multiplexes vCPUs across physical CPU cores
6
Virtual NIC → hypervisor software bridge → physical NIC → network traffic flows normally
VM vs Container vs Bare Metal
ApproachIsolationOverhead
Bare metalComplete — one OS per machineNone — maximum performance
Type 1 VMStrong — separate kernel per guest~3-5% for CPU; higher for I/O
Type 2 VMStrong — separate kernel per guest10-20% — extra host OS layer
ContainerWeaker — shared host kernel~1-2% — near-native performance
Real-World Example
University CS Lab (Type 2):
  Host: Windows 11 with VirtualBox
  Each student's VM: Ubuntu 22.04, 2 vCPUs, 4 GB RAM, 40 GB virtual disk
  20 VMs on one server: 40 cores, 128 GB RAM physical host
  
AWS EC2 (Type 1 — Nitro Hypervisor):
  Your t3.medium instance: 2 vCPUs, 4 GB RAM
  Physical host might run 50+ customer instances simultaneously
  Complete isolation: you cannot see or affect other customers' VMs
  
Docker Container (NOT a VM):
  Same Linux kernel as host — just isolated user space
  Container starts in <1 second (vs 30-60 seconds for a VM)
  10x lower memory overhead than an equivalent VM
  Trade-off: if host kernel has a bug, ALL containers are affected
Exam Pitfalls
✕
Containers are NOT VMs. Containers share the host OS kernel. VMs include a full guest OS with their own kernel. This is the most critical distinction in virtualisation questions.
✕
Java JVM is NOT a hypervisor-style VM. JVM virtualises a programming language runtime, not hardware. It has nothing to do with OS virtualisation.
✕
Para-virtualisation requires modifying the guest OS. Unmodified Windows cannot be para-virtualised — it requires OS source code changes.
✕
Live migration requires shared storage (SAN/NAS). The VM disk must be accessible from both source and destination hosts simultaneously.
✕
Type 1 ≠ always more secure than Type 2. Type 1 has fewer software layers but a larger, more complex attack surface on the hypervisor itself.

OS Design & Implementation

Goals, principles, and how modern OSes are actually built

Core Concept
OS design involves defining goals (user goals vs system goals), choosing structural strategies (monolithic/microkernel/hybrid), and implementing mostly in C/C++ with a small assembly core for hardware-specific operations. The key design principle is separating policy (what to do) from mechanism (how to do it).
Key Terminology
Policy
WHAT should be done — "Give CPU to the highest-priority process"
Mechanism
HOW to do it — the priority queue data structure + context switch code
Policy/Mechanism Separation
Key design principle: change scheduling policy without rewriting the scheduler mechanism
Portability
Ability to run the OS on different hardware architectures with minimal code changes
HAL
Hardware Abstraction Layer — isolates OS code from hardware-specific details
User Goals
Convenient, easy to use, reliable, safe, fast — what end users care about
System Goals
Easy to design/implement/maintain, flexible, error-free, efficient — what developers care about
UNIX Philosophy
Small focused programs that do one thing well and communicate via pipes
Design Process — Steps
1
Define goals: User goals (fast, easy) vs System goals (maintainable, efficient) — these often conflict
2
Choose structure: Monolithic for performance (Linux), Microkernel for reliability (Minix), Hybrid for balance (Windows, macOS)
3
Separate policy from mechanism: Timer interrupt = mechanism. How long to run each process = policy (changeable)
4
Write in high-level language (C/C++): Provides portability; ~95% of modern OS code is C
5
Write hardware-specific code in assembly: Context switch, interrupt vectors, boot sequence — architecture-specific
6
Test on emulators/VMs first: Debug kernel on QEMU before running on real hardware
7
Profile and optimise: Find bottlenecks (usually in scheduler, memory allocator, I/O paths)
Design Trade-offs
DecisionAdvantageCost
C instead of AssemblyPortable across x86, ARM, RISC-V, MIPSSlight performance overhead vs hand-tuned assembly
Policy/Mechanism separationSwap scheduling policy without rewriting schedulerMore complex architecture; harder to optimise
Monolithic designFast — no IPC between OS componentsAny bug can crash the whole system
Microkernel designFault-isolated; safer; easier to verify formallyIPC overhead; slower for I/O-intensive workloads
Example — Linux vs Windows NT design
Linux Design:
  • ~95% C, ~5% architecture-specific assembly
  • Monolithic kernel + loadable modules (compromise)
  • Policy/mechanism separation: CFS scheduler mechanism supports
    SCHED_NORMAL, SCHED_FIFO, SCHED_RR policies — same code, different behaviour
  • Portability: same source compiles for x86, ARM, RISC-V, PowerPC
    Only arch/ directory is architecture-specific

Windows NT Design:
  • HAL layer: same NT kernel binary runs on x86 and ARM (via HAL swap)
  • Hybrid kernel: Executive in kernel mode; Win32 subsystem partly in user
  • Written in C and C++; Security Reference Monitor enforces all access checks
  • Object Manager: every kernel resource (files, processes, threads) is an object

Key policy/mechanism example:
  Mechanism: "context switch every timer interrupt"
  Policy: "quantum = 20ms for interactive, 60ms for batch"
  Change policy → different scheduler class. Mechanism unchanged.
Exam Pitfalls
✕
Policy and mechanism MUST be separated. Classic exam question: "The timer interrupt mechanism fires every 10ms — the policy decides how many ms each process gets." Don't confuse the two.
✕
OS is NOT written entirely in assembly. Modern OSes are primarily C (95%+). Only boot/context switch/interrupt code uses assembly.
✕
BIOS and UEFI are firmware, not the OS. They are permanently stored in ROM/flash and run before the OS loads from disk.
✕
Efficiency and user-friendliness CONFLICT. An OS optimised for maximum performance is harder to use. This trade-off must be acknowledged in design questions.

Shell Programming

Commands, scripts, and automation on Unix/Linux systems

Core Concept
A shell is a command interpreter that translates user commands into OS system calls. Shell scripting combines these commands with variables, control flow (if/for/while), and functions to automate tasks — creating executable programs without compilation.
Key Terminology
bash
Bourne Again SHell — default shell on most Linux distributions; superset of sh
Shebang (#!)
First line of script declaring interpreter: #!/bin/bash — must be at column 1
Variable
NAME=value (NO spaces around =). Read with $NAME or ${NAME}
Positional Parameters
$0=script name, $1..$9=arguments, $#=count, $@=all args, $*=all as one string
Exit Code ($?)
Return value of last command: 0 = SUCCESS, non-zero = ERROR
Pipe (|)
Connects stdout of left command to stdin of right command
Redirect (> and >>)
> overwrites file; >> appends to file; 2> redirects stderr; &> both
$() — Command Substitution
Runs a command and captures output: DATE=$(date +%Y%m%d)
Cron
System scheduler for recurring shell scripts: 0 2 * * * /backup.sh = run daily at 2 AM
Stdin/Stdout/Stderr
File descriptors 0 (input), 1 (output), 2 (error) — can be redirected independently
Mechanism — Writing and running a script
1
Create file: nano backup.sh
2
Add shebang on line 1: #!/bin/bash — tells OS which interpreter to use
3
Declare variables: SRC="/home/alice" (no spaces around =)
4
Add conditionals: if [ -d "$SRC" ]; then ... fi
5
Add loops: for f in *.log; do echo "$f"; done
6
Make executable: chmod +x backup.sh
7
Run: ./backup.sh or bash backup.sh
8
Check exit code: echo $? — 0 = success

Complete Example Script
#!/bin/bash
# ─── Backup Script with error handling ───────────────────

SRC_DIR="/home/alice/documents"
BACKUP_DIR="/backup"
DATE=$(date +%Y%m%d_%H%M%S)          # command substitution
ARCHIVE="$BACKUP_DIR/docs_$DATE.tar.gz"

# ── Check source exists ──
if [ ! -d "$SRC_DIR" ]; then
    echo "ERROR: Source directory not found!" >&2   # stderr
    exit 1                                           # non-zero = failure
fi

# ── Create backup dir if needed ──
[ ! -d "$BACKUP_DIR" ] && mkdir -p "$BACKUP_DIR"

# ── Perform backup ──
echo "Backing up $SRC_DIR → $ARCHIVE"
tar -czf "$ARCHIVE" "$SRC_DIR"

# ── Check success ──
if [ $? -eq 0 ]; then
    echo "SUCCESS: $(du -sh "$ARCHIVE" | cut -f1) saved"
else
    echo "FAILED: tar returned error" >&2
    exit 2
fi

# ── Delete backups older than 7 days ──
find "$BACKUP_DIR" -name "*.tar.gz" -mtime +7 -delete

Exam Pitfalls
✕
No spaces around = in variable assignment. NAME=Alice ✓ vs NAME = Alice ✗ (treats NAME as a command with = and Alice as arguments)
✕
Exit code 0 = SUCCESS — counterintuitive for programmers. In most languages 0 = false. In shell exit codes, 0 = true/success.
✕
Always quote variables: "$VAR" — unquoted variables break silently on filenames with spaces. Use double quotes consistently.
✕
[ ] vs [[ ]]: Single brackets are POSIX sh. Double brackets are bash-specific but safer for string comparisons. [[ "$a" == "$b" ]] handles empty strings safely.
✕
chmod +x does NOT make a file a valid script. It only marks it executable. The file still needs a valid shebang and correct bash syntax.
✕
$() is preferred over backticks for command substitution — it nests cleanly and is more readable: DIR=$(dirname $(which bash)) works; nested backticks do not.
